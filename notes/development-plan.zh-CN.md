# MediaScribe 路线图

这是一份基于当前实现状态整理的精简路线图。

## 当前状态

已完成：
- 项目名称与 CLI 命名已经统一为 `MediaScribe` / `mediascribe`
- 源码已经收敛到单一包目录 `mediascribe/`
- 音频、文本、视频流程已经拆分为可复用服务
- ASR 路由已支持本地与云端 provider
- 默认本地总结路径已接入 Ollama，默认模型为 `ollama/qwen2.5:3b`
- 中英文 README / quickstart 已拆分为独立文档
- MIT License 和包元数据已补齐
- 仓库中的缓存、构建残留和运行输出已完成一轮清理

## 产品形态

MediaScribe 当前应保持为一个聚焦型 CLI，覆盖：
- 音频转写
- transcript 总结
- 文本总结
- 视频总结

核心工作流原则：
- 适合先转写、后总结
- 视频优先走字幕路径
- 默认优先本地方案，必要时再走云端
- CLI 保持简单，底层服务保持可复用

## 近期路线

### 1. 稳定性

- 保持主 CLI 流程稳定、可预期
- 持续补强音频、文本、视频路径的自动化覆盖
- 保持 provider 路由逻辑清晰、易排查

### 2. 本地优先可用性

- 让 Ollama 本地总结路径对新用户更顺手
- 保持本地硬件说明清晰、实用
- 改善环境检查和初始化提示

### 3. 文档质量

- `docs/` 只放面向用户的正式文档
- `notes/` 只放规划、草稿和归档材料
- 示例与当前“本地默认总结”路线保持一致

### 4. 运维体验

- 优化各 provider 的错误提示
- 在确实能降低配置门槛的地方增加或完善 doctor / 诊断命令
- 对 Windows 常见使用路径给出更明确说明

## 优先候选项

- 轻量环境自检能力，覆盖 summary / ASR 前置条件
- 更明确的 Ollama 连接失败提示
- 更多面向本地总结流程的端到端小示例
- 补充本地总结模型的 benchmark 说明

## 暂不优先

- GUI 应用
- 实时流式处理
- 过重的框架抽象
- 大范围插件市场化扩展
- 没有明确需求前提下的多进程编排

## 发布前检查

发布前建议确认：
- README 与 quickstart 的中英文版本一致
- CLI help 文案与示例命令正确
- 测试套件通过
- `.gitignore` 已覆盖本地个人文件
- License 与包元数据完整
- 仓库再次做一轮清理
